# ADR-0028: PROMOTION_CHECKLIST Universality Criteria

**Status**: Accepted
**Date**: 2026-06-05
**Deciders**: architect
**Supersedes**: —

## Context

`PROMOTION_CHECKLIST.md` was introduced during the co-safety L2 variant creation as the gate an L2 project must pass before being promoted to an L1 variant template. The 7 conditions were derived inductively from co-safety's specific requirements.

Before applying the checklist to future variants, the conditions must be classified as either **Universal** (enforced identically for all variants) or **Domain-specific** (each variant defines its own criteria, overriding the default). Without this classification, future variant authors may apply co-safety-specific rules to unrelated domains, causing false failures.

**Platform Impact**: Both. `PROMOTION_CHECKLIST.md` is generated by `create-l2-scaffold.ts` (L0-only script) but resides in the L2 project output; the check applies to all platforms where L2 projects are created and reviewed.

## Decision

### Condition Classification

| # | Condition | Class | Notes |
|---|-----------|-------|-------|
| 1 | All agents registered in `AGENTS.md` and agent files exist | **Universal** | Required by `validate-templates.ts` for all variants |
| 2 | Domain skills validated (skill names, triggers, format) | **Domain-specific** | Each variant defines its own expected skill list |
| 3 | Domain artifacts complete with all required fields | **Domain-specific** | Required fields are domain-defined (e.g., incident fields for EHS) |
| 4 | Domain audit script passes (`bun scripts/audit-<domain>.ts`) | **Domain-specific** | Each variant names and owns its audit script |
| 5 | Platform parity: `CLAUDE.md` ↔ `GEMINI.md` content aligned | **Universal** | Enforced by `validate-templates.ts` P-01 check for all variants |
| 6 | `bun install` completes and `setup.sh` runs without error | **Universal** | Required for all variants regardless of domain |
| 7 | Human review of `_ORIGIN.md` § Manual Phase B Steps | **Universal** | All variants have manual promotion steps that require human sign-off |

### Rule

- **Universal conditions (1, 5, 6, 7)** must pass unchanged for every variant. They may not be overridden in a variant's `PROMOTION_CHECKLIST.md`.
- **Domain-specific conditions (2, 3, 4)** must be redefined by each variant. The L2 scaffold template provides placeholder text that variant authors replace. An unmodified placeholder is a checklist failure.

### Enforcement

`create-l2-scaffold.ts` generates `PROMOTION_CHECKLIST.md` with Universal conditions pre-filled and Domain-specific conditions as clearly marked `TODO:` blocks. A future lint rule may flag unmodified TODOs as promotion blockers.

## Consequences

**Positive:**

- Variant authors know exactly which conditions are non-negotiable vs. customizable
- Reduces false positives when applying the checklist to non-EHS domains
- Universal conditions provide a stable, testable baseline across all variants

**Negative / Trade-offs:**

- Domain-specific conditions require variant authors to actively define requirements — more upfront work per variant
- Classification is based on co-safety experience alone; a future variant may surface edge cases requiring reclassification

**Future Work:**

- [ ] Add a lint rule to `bun scripts/audit.ts` that flags unmodified TODO blocks in `PROMOTION_CHECKLIST.md`
- [ ] Re-evaluate classification after 3+ variants have been promoted through the checklist
