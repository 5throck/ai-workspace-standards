### 6.5 Script Lifecycle Management {#script-lifecycle-management}

Scripts are managed across three ownership layers following the SSOT architecture. Development originates at Tier 1 (Workspace Root) and is explicitly published to Tier 2 (Templates).

#### Ownership Layers

| Layer | Location | Owner | Update Policy |
|-------|----------|-------|---------------|
| **Tier 1 — Workspace Root (SSOT)** | `scripts/` (workspace root) | workspace maintainer | Versioned via `scripts/SCRIPTS.md` |
| **Tier 2 — Templates** | `templates/common/scripts/` | publish: `bun run propagate:apply` | Explicit publish from Tier 1 |
| **Tier 3 — New Project** | `<project>/scripts/` | project team | Independent snapshot after creation |

**Propagation rule**: Development happens at Tier 1 (workspace `scripts/`). Publish Tier 1→Tier 2 explicitly using `bun run propagate:apply` before creating new projects. Tier 3 is a snapshot of Tier 2 at project creation time — no automatic back-propagation from Tier 3.

#### Script States

| Status | Meaning | Enforcement |
|--------|---------|-------------|
| `active` | In production use | Version bump required on change |
| `deprecated` | Scheduled for removal | `removal-date` required (90-day minimum notice); `dev-sync` warns Tier 2/Tier 3 consumers |
| `experimental` | Unstable / not propagated | Not synced to Tier 2/Tier 3 automatically |

#### Dependency Tracking

Scripts that call other scripts must declare dependencies in `SCRIPTS.md` Registry:

```markdown
| script | source | version | status | removal-date | security-advisory | depends_on |
|--------|--------|---------|--------|--------------|-------------------|------------|
| `audit.ts` | Tier 1 | 1.2.0 | active | — | — | — |
| `dev-sync.ts` | Tier 1 | 2.0.0 | active | — | — | `audit.ts` |
```

The `verify-scripts.ts` script checks for:
- ✅ Circular dependencies (A depends on B, B depends on A)
- ✅ Missing dependencies (declared dependency doesn't exist)
- ✅ Stale dependencies (declared dependency has been deprecated)

#### Deprecation Protocol

1. Set `status: deprecated` and `removal-date: YYYY-MM-DD` in `SCRIPTS.md` Registry (≥ 90 days notice)
2. `dev-sync.ts` warns all Tier 2/Tier 3 consumers on every run until removed
3. On `removal-date`, `verify-scripts.ts --verify` **hard blocks** pre-commit

#### Security Advisory Protocol

When a security vulnerability is found in a script:
1. Set `security-advisory: CVE-XXXX` in `SCRIPTS.md` Registry
2. `dev-sync.ts` **hard blocks immediately** in Tier 2/Tier 3 (no grace period)
3. Advisory is cleared only after the script is patched or removed

#### SCRIPTS.md — Registry Format

`scripts/SCRIPTS.md` is the Tier 1 manifest. It has two sections:

- **`## Registry`** — machine-readable table parsed by `verify-scripts.ts`
- **`## Guide`** — human-readable documentation (purpose, usage, deprecation notes)

```markdown
| script | source | version | status | removal-date | security-advisory | depends_on |
|--------|--------|---------|--------|--------------|-------------------|------------|
| `audit.sh` | Tier 1 | 1.2.0 | active | — | — | — |
| `old-tool.ts` | Tier 1 | 2.1.0 | deprecated | 2026-08-01 | — | — |
| `dev-sync.ts` | Tier 1 | 2.0.0 | active | — | — | `audit.sh` |
```
