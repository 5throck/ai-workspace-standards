### 6.5 Script Lifecycle Management {#script-lifecycle-management}

Scripts are managed across three ownership layers following the SSOT architecture. Development originates at Tier 1 (Workspace Root) and is explicitly published to Tier 2 (Templates).

#### Ownership Layers

| Layer | Location | Owner | Update Policy |
|-------|----------|-------|---------------|
| **Tier 1 — Workspace Root (SSOT)** | `scripts/` (workspace root) | workspace maintainer | Versioned via `scripts/SCRIPTS.md` |
| **Tier 2 — Templates** | `templates/common/scripts/` | publish: `bun scripts/propagate-to-templates.ts --apply` | Explicit publish from Tier 1 |
| **Tier 3 — New Project** | `<project>/scripts/` | project team | Independent snapshot after creation |

**Propagation rule**: Development happens at Tier 1 (workspace `scripts/`). Publish Tier 1→Tier 2 explicitly using `bun scripts/propagate-to-templates.ts --apply` before creating new projects. Tier 3 is a snapshot of Tier 2 at project creation time — no automatic back-propagation from Tier 3.

> **Tool**: `propagate-to-templates.ts` v2.0.2 (L0-only). Replaces the deleted `publish-to-template.ts`.
> **Flags**: `--apply`, `--dry-run`, `--prune`, `--check-drift`, `--governance-l1`, `--docs`

#### Script States

| Status | Meaning | Enforcement |
|--------|---------|-------------|
| `active` | In production use | Version bump required on change |
| `deprecated` | Scheduled for removal | `removal-date` required (90-day minimum notice); `dev-sync` warns Tier 2/Tier 3 consumers |
| `experimental` | Unstable / not propagated | Not synced to Tier 2/Tier 3 automatically |

#### Dependency Tracking

Scripts that call other scripts must declare dependencies in `SCRIPTS.md` Registry:

```markdown
| script | source | version | status | removal-date | security-advisory | depends_on |
|--------|--------|---------|--------|--------------|-------------------|------------|
| `audit.ts` | Tier 1 | 1.2.0 | active | — | — | — |
| `dev-sync.ts` | Tier 1 | 2.0.0 | active | — | — | `audit.ts` |
```

The `verify-scripts.ts` script checks for:
- ✅ Circular dependencies (A depends on B, B depends on A)
- ✅ Missing dependencies (declared dependency doesn't exist)
- ✅ Stale dependencies (declared dependency has been deprecated)

#### Deprecation Protocol

1. Set `status: deprecated` and `removal-date: YYYY-MM-DD` in `SCRIPTS.md` Registry (≥ 90 days notice)
2. `dev-sync.ts` warns all Tier 2/Tier 3 consumers on every run until removed
3. On `removal-date`, `verify-scripts.ts --verify` **hard blocks** pre-commit

#### Security Advisory Protocol

When a security vulnerability is found in a script:
1. Set `security-advisory: CVE-XXXX` in `SCRIPTS.md` Registry
2. `dev-sync.ts` **hard blocks immediately** in Tier 2/Tier 3 (no grace period)
3. Advisory is cleared only after the script is patched or removed

#### Variant-Specific Scripts (L2 Layer)

Variant templates (`templates/co-*/`) may include scripts that are specific to that variant's workflow and are not part of the shared L1 common scripts.

**Canonical placement rule**: Variant scripts MUST be placed in `scripts/<variant>/` (a subdirectory), NOT in the top-level `scripts/` directory.

```text
<project>/
└── scripts/
    ├── audit.ts            ← L1 common (top-level, registered in SCRIPTS.md)
    ├── dev-sync.ts         ← L1 common (top-level, registered in SCRIPTS.md)
    └── co-deck/            ← L2 variant-specific (subdirectory)
        ├── SCRIPTS.md      ← variant registry (separate from L1 SCRIPTS.md)
        ├── snapshot.ts
        └── download-font.ts
```

**Why subdirectory?** — Audit system constraint (intentional design):

The L1 `audit.ts` function `verifyScriptRegistryConsistency()` uses a **non-recursive** `readdirSync(scriptsDir)` call:
```typescript
// Only top-level .ts files are checked
const scripts = fs.readdirSync(scriptsDir).filter(f => f.endsWith('.ts'));
```
Top-level `.ts` files must be registered in `scripts/SCRIPTS.md`. Variant scripts in `scripts/<variant>/` are in a subdirectory and therefore excluded from this check. This exclusion is **intentional** — variant scripts are not registered in the shared L1 SCRIPTS.md.

> ⚠️ Do NOT change `readdirSync` to recursive without updating the variant-scripts-in-subdirectory pattern throughout all variants.

**L1 `scripts/SCRIPTS.md` must not be overwritten**: A variant MUST NOT include a `scripts/SCRIPTS.md` at the top-level `scripts/` path in its overlay. If it does, it overwrites the L1 SCRIPTS.md during scaffolding, causing `verifyScriptRegistryConsistency()` to fail for all 30+ L1 scripts. The variant's own script registry belongs at `scripts/<variant>/SCRIPTS.md`.

**`variant.json` declaration required**: Any variant with `script_manifest.local` entries must declare them in `variant.json`. `bun scripts/validate-templates.ts` verifies that each declared path exists (check B-03). See [ADR-0033](../adr/0033-variant-specific-skills-scripts-blueprint.md).

**Invocation**: `bun scripts/<variant>/<name>.ts [args]` or via `package.json` npm scripts.

---

#### SCRIPTS.md — Registry Format

`scripts/SCRIPTS.md` is the Tier 1 manifest. It has two sections:

- **`## Registry`** — machine-readable table parsed by `verify-scripts.ts`
- **`## Guide`** — human-readable documentation (purpose, usage, deprecation notes)

```markdown
| script | source | version | status | removal-date | security-advisory | depends_on |
|--------|--------|---------|--------|--------------|-------------------|------------|
| `audit.ts` | Tier 1 | 1.2.0 | active | — | — | — |
| `old-tool.ts` | Tier 1 | 2.1.0 | deprecated | 2026-08-01 | — | — |
| `dev-sync.ts` | Tier 1 | 2.0.0 | active | — | — | `audit.ts` |
```

#### Scope Legend

| Scope | Meaning |
|-------|---------|
| `L0` | workspace root only; must NOT be copied to `templates/common/` or L2 projects |
| `L0+L1` | exists in `scripts/` AND `templates/common/scripts/`; scaffold-copies to L2 at `new-project` time |
| `L0+L1+L2` | reserved for future use (Fork Model architecture — not currently used) |

#### L2 Propagation Control

Scripts with scope `L0` in SCRIPTS.md are excluded at the **L0→L1 propagation stage** — `propagate-to-templates.ts` reads the scope via `helpers/layer-filter.ts` and skips these scripts entirely. They never enter `templates/common/scripts/` and therefore never reach L2 projects.

To mark a script as workspace-only, set its scope to `L0` in SCRIPTS.md. No file-header annotation is required.

**Current L0-only scripts (examples)**: `upgrade-project.ts`, `new-project.ts`, `propagate-to-templates.ts`

#### Tier 3 SCRIPTS.md Filtering {#tier-3-script-md-filtering}

When a Tier 3 project is scaffolded (via `new-project.ts` or `create-l2-scaffold.ts`), the L1 `scripts/SCRIPTS.md` is copied into the project. This file contains the full L0+L1 registry, including rows for L0-only scripts that are intentionally excluded from the project. Two mechanisms ensure consistency:

| Stage | Mechanism | Tool | Scope |
|-------|-----------|------|-------|
| **L0→L1 file filtering** | `.ts` files with `layer: L0` are not propagated to `templates/common/scripts/` | `propagate-to-templates.ts` via `layer-filter.ts` | Prevents L0-only scripts from reaching L1 |
| **L1→L2 row filtering** | Registry rows with `layer: L0` are stripped from the project's `scripts/SCRIPTS.md` | `new-project.ts` §6.5, `create-l2-scaffold.ts` `copyCommonOverlay()` | Ensures SCRIPTS.md matches on-disk files |
| **Runtime defense-in-depth** | `verify-scripts.ts` v1.3.0+ auto-detects L0/L1/L2 context and dynamically skips L0-only entries | `verify-scripts.ts` via `detectContextLayer()` + `isLayerRelevant()` | Catches any residual filtering gaps |

**What gets rewritten**: The project's SCRIPTS.md header is changed from "Tier 1 SSOT" to "Tier 3 project snapshot" to clearly indicate its filtered status. The `source` column for project-local scripts should use `L2` (not `L0`).

**Why dual filtering?**: Row-level filtering at scaffold time produces a clean SCRIPTS.md that matches the on-disk file inventory. Runtime filtering in `verify-scripts.ts` provides defense-in-depth — if a future scaffolding path misses the filtering step, verification still passes without false-positive ghost entry errors.

#### SCRIPTS.md in L2 Projects

L2 projects receive a **filtered snapshot** of the L1 SCRIPTS.md at scaffold time. Key rules:

| Aspect | Rule |
|--------|------|
| **L0-only rows** | Stripped at scaffold time (both `new-project.ts` and `create-l2-scaffold.ts`) |
| **Header text** | Rewritten to "project snapshot" — not SSOT |
| **Project-local scripts** | Registered with `source: L2`, `layer: L0+L1` (not `L0`) |
| **Sub-directory scripts** | Tracked in `scripts/<variant>/SCRIPTS.md` sub-registry, not the main registry |
| **Version bumps** | Independent after scaffolding — no back-propagation to L1 |

**Adding a new script to an L2 project**: Register it in `scripts/SCRIPTS.md` with `source: L2` and `layer: L0+L1`. If the script is variant-specific, place it in `scripts/<variant>/` and register it in `scripts/<variant>/SCRIPTS.md` instead.

#### L0+L1 Script Runtime Context Guard Convention

Scripts with scope `L0+L1` exist in both the workspace root (`scripts/`) and `templates/common/scripts/`. When a new project is scaffolded, these scripts are copied into the L2 project. At that point, `CONSTITUTION.md` is **not** present — L2 projects use `docs/context.md` instead (see [CONSTITUTION.md §7.5](../../CONSTITUTION.md) — *CONSTITUTION.md Non-Propagation*).

**All `L0+L1` scripts must gracefully handle L1/L2 context.** Follow this convention:

##### 1. Canonical L0 Detection Pattern

```typescript
const CONSTITUTION_FILE = join(ROOT, 'CONSTITUTION.md');
const IS_WORKSPACE_ROOT = existsSync(CONSTITUTION_FILE);
```

- **Always use `existsSync('CONSTITUTION.md')`** as the canonical L0 detection signal.
- Do **not** use alternative heuristics (`existsSync('templates/common')`, file path checks, etc.) unless explicitly documented and justified.
- Name the variable `IS_WORKSPACE_ROOT` for consistency across scripts.

##### 2. Gate L0-Only Logic

Any logic that requires workspace-root infrastructure (e.g., `AGENTS.md` registry, `scripts/SCRIPTS.md`, `templates/` hierarchy, L0→L1 publish) must be guarded:

```typescript
function runWorkspaceOnlyCheck(): Issue[] {
  const issues: Issue[] = [];
  if (!IS_WORKSPACE_ROOT) return issues;  // ← early return
  // ... L0-only logic ...
  return issues;
}
```

**Examples of L0-only operations that require gating**:
- Checking `AGENTS.md` registry consistency (no `AGENTS.md` in L2)
- Comparing scripts/ against SCRIPTS.md — in L2, SCRIPTS.md is a filtered snapshot; `verify-scripts.ts` v1.3.0+ auto-detects context and skips L0-only entries (see [Tier 3 SCRIPTS.md Filtering](#tier-3-script-md-filtering) below)
- Scanning `templates/common/` for sync drift (no `templates/` in L2)
- L0→L1 publish (`propagate-to-templates.ts`)
- L0-only content removal verification

##### 3. Guard L0-Only Sub-Script Calls

When an `L0+L1` script invokes another script that may only exist at L0, wrap the call with an `existsSync` guard:

```typescript
const auditTs = path.join('scripts', 'audit.ts');
if (fs.existsSync(auditTs)) {
  const result = await $`bun scripts/audit.ts`.nothrow();
  // ... handle result ...
}
```

**Scripts that may not exist in L2**: `audit.ts` (L0 workspace root), `sync-md.ts` (L0 workspace root), `propagate-to-templates.ts` (L0-only by scope).

##### 4. Directory Scan Scope Limitation

When scanning directories recursively at the project root, always restrict the scan to known subdirectories — do not recurse into arbitrary directories:

```typescript
// CORRECT: Always restrict root-level scan (L0 and L1/L2 alike)
if (dir === ROOT) {
  if (entry.name !== 'skills' && entry.name !== '.claude') continue;
}
```

This prevents noise and false positives when the script runs in an L1/L2 project that may contain unrelated directories (`src/`, `docs/`, etc.).

##### 5. Avoid Shadow Variables

Do not introduce loop-local variables that reuse the name `isWorkspaceRoot` or `IS_WORKSPACE_ROOT` with different semantics. Use a distinct name (e.g., `isWorkspaceRootReadme`) that composes with the module-level `IS_WORKSPACE_ROOT`:

```typescript
// WRONG: shadow variable ignores IS_WORKSPACE_ROOT
const isWorkspaceRoot = relPath === 'README.md';

// CORRECT: composes with module-level detection
const isWorkspaceRootReadme = IS_WORKSPACE_ROOT && relPath === 'README.md';
```
