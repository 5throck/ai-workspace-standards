# Co-Security 템플릿

AI 기반 보안 인게이지먼트 워크스페이스 템플릿. 구조화된 멀티 에이전트 아키텍처를 통해 레드 팀 운영, 위협 모델링, 크로스 플랫폼 패치 자동화에 최적화되어 있습니다.

## 개요

co-security 템플릿은 전체 공격/방어 사이클에 걸쳐 AI 어시스턴트를 활용하는 보안 인게이지먼트를 위한 즉시 사용 가능한 워크스페이스를 제공합니다. Claude Code(`CLAUDE.md`)와 스코핑부터 검증까지 전 과정을 아우르는 구조화된 멀티 에이전트 시스템을 결합하여, 모든 인게이지먼트에 첫날부터 일관된 명령, 에이전트 역할, 권한 부여 게이트, 품질 관리를 제공합니다.

주요 특징:

- **권한 부여 게이트 워크플로우** — Phase 1 이후 활동을 시작하기 전에 `verify-authorization` 스킬이 반드시 통과되어야 합니다. 서명된 권한 부여 문서 없이는 정찰, 익스플로잇, 패치 작업이 진행될 수 없습니다.
- **멀티 에이전트 아키텍처** — 전문화된 에이전트(보안 PM, 레드 팀 리드, 펜테스터, 위협 모델러, 패치 엔지니어, 리포트 작성자)가 각 인게이지먼트 단계를 담당하여 공격 및 복구 작업 간 컨텍스트 혼입을 방지합니다.
- **전 단계 커버리지** — 스코핑부터 검증/재테스트까지 6단계로 구성되며, 각 게이트에서 명확한 핸드오프 기준을 제공합니다.
- **크로스 플랫폼 패치 자동화** — 복구 단계에서 `ansible/` 디렉터리에서 관리되는 SSH 및 Ansible 플레이북을 통해 Windows, macOS, Linux 대상을 지원합니다.
- **슬래시 커맨드 기반** — 일반적인 작업이 슬래시 커맨드로 노출되어 일상적인 인게이지먼트 업무가 빠르고 감사 가능하며 일관성 있게 유지됩니다.

> **상태:** 베타 (v0.2.0) — 핵심 워크플로우는 안정적이나, 일부 에이전트 역할과 Ansible 모듈은 활발히 개선 중입니다.

## 빠른 시작

```bash
bash scripts/new-project.sh "engagement-name" --variant co-security
```

이 명령은 워크스페이스 루트 아래에 새 인게이지먼트 디렉터리를 생성하고, 모든 co-security 템플릿 파일을 복사하며, 초기 변수 치환(인게이지먼트 이름, 날짜, 클라이언트 플레이스홀더)을 수행합니다.

스캐폴딩 이후:

1. 편집기에서 새 인게이지먼트 디렉터리를 엽니다.
2. `docs/scope.md`에 대상 시스템, 교전 규칙, 스코프 외 경계를 작성합니다.
3. 서명된 권한 부여 문서를 `docs/authorization.md`에 저장합니다.
4. `/sync "chore: initial scaffold"`를 실행하여 첫 번째 커밋과 PR을 생성합니다.
5. Phase 1을 시작하기 전에 `verify-authorization` 스킬을 실행합니다.

## 권한 부여 게이트

> **이 게이트는 협상 불가입니다.** `verify-authorization` 스킬은 `docs/authorization.md`가 존재하고, 내용이 비어 있지 않으며, 필수 필드(스코프 해시, 권한 부여자, 만료일)를 포함하는지 확인합니다. Phase 1 이후 활동을 시작하는 모든 에이전트 또는 커맨드는 반드시 이 스킬을 먼저 실행해야 하며, 실패 시 즉시 중단해야 합니다.

```
verify-authorization          # 정찰, 익스플로잇, 패치 작업 시작 전 실행
```

권한 부여가 없거나 만료된 경우, 문서가 업데이트되고 스킬이 통과될 때까지 모든 공격 활동을 중단해야 합니다.

## 단계별 워크플로우

| 단계 | 이름 | 설명 | 게이트 |
|------|------|------|--------|
| **0** | 스코핑 | 대상, 교전 규칙, 성공 기준 정의 | 서명된 스코프 문서 |
| **1** | 정찰 | 공격 표면의 수동/능동 정찰 | `verify-authorization` 통과 |
| **2** | 위협 모델링 | STRIDE/PASTA 분석, 공격 트리 구성 | Phase 1 결과 커밋 |
| **3** | 익스플로잇 | 스코프 내 취약점에 대한 통제된 익스플로잇 | 위협 모델 PM 승인 |
| **4** | 복구 | 패치 개발 및 크로스 플랫폼 배포 | 익스플로잇 리포트 커밋 |
| **5** | 리포팅 | CVSS 점수 및 복구 단계 포함 전체 인게이지먼트 리포트 | 모든 결과물이 `docs/findings/`에 존재 |
| **6** | 검증 | 패치된 시스템 재테스트, 모든 결과물 종료 확인 | `PATCH_LOG.md` 패치 로그 완료 |

## 프로젝트 구조

```
<engagement-name>/
├── CLAUDE.md                   # Claude Code 동작 설정
├── CONSTITUTION.md             # 공유 워크스페이스 표준 (루트에서 심볼릭 링크)
├── AGENTS.md                   # 에이전트 로스터 및 디스패치 규칙
├── CHANGELOG.md                # 미출시 및 버전별 변경 이력
├── PATCH_LOG.md                # 적용된 모든 패치의 시간순 로그
├── agents/                     # 에이전트별 역할 정의 파일
│   ├── pm.md                   # 보안 PM — 오케스트레이션 및 게이트 집행
│   ├── red-team-lead.md        # 레드 팀 리드 — 공격 전략 및 Phase 3 감독
│   ├── pentester.md            # 펜테스터 — 직접 익스플로잇 및 PoC 개발
│   ├── threat-modeler.md       # 위협 모델러 — STRIDE/PASTA, 공격 트리
│   ├── patch-engineer.md       # 패치 엔지니어 — 복구 스크립트 및 Ansible 플레이북
│   └── report-writer.md        # 리포트 작성자 — 결과물 문서화 및 경영진 요약
├── ansible/                    # 크로스 플랫폼 패치 배포를 위한 Ansible 플레이북
│   ├── inventory/              # 호스트 인벤토리 (Windows, macOS, Linux)
│   ├── playbooks/              # 결과물별 복구 플레이북
│   └── roles/                  # 재사용 가능한 Ansible 역할
├── docs/                       # 인게이지먼트 문서
│   ├── authorization.md        # 서명된 권한 부여 문서 (Phase 1 전 필수)
│   ├── scope.md                # 대상 시스템, 교전 규칙, 스코프 외 항목
│   └── findings/               # 개별 결과물 파일 (취약점 1건당 1파일)
├── memory/                     # 일일 세션 로그 (YYYY-MM-DD.md)
└── scripts/                    # 유틸리티 스크립트 (audit.sh, dev-sync.sh, …)
```

## 사용 가능한 커맨드

| 커맨드 | 설명 |
|--------|------|
| `/sync "feat: ..."` | 전체 파이프라인 — memlog → sync-md → changelog → audit → commit → PR |
| `/changelog "..."` | `CHANGELOG.md`의 `[Unreleased]` 항목에 내용 추가 |
| `/memlog "summary"` | 오늘의 메모리 로그에 세션 요약 추가 |
| `/new-task "name"` | 오늘의 메모리 로그에 작업 추적 블록 생성 |
| `/security-check` | 보안 권고 스캔 실행 (일일 또는 `--pr` PR 전 모드) |

모든 커맨드는 `.claude/commands/`에 정의되어 있으며 Claude Code에 스킬로 자동 등록됩니다.

## 스킬

| 스킬 | 단계 | 설명 |
|------|------|------|
| `verify-authorization` | 게이트 | 공격 활동 전 `docs/authorization.md` 유효성 검사 |
| `engagement-scoping` | 0 | `docs/scope.md` 작성 및 단계 기준 설정 지원 |
| `recon-surface` | 1 | 공격 표면 열거 및 수동/능동 정찰 안내 |
| `threat-modeling` | 2 | STRIDE/PASTA 분석 실행 및 공격 트리 생성 |
| `finding-tracker` | 3–4 | CVSS 점수와 함께 `docs/findings/`에 결과물 파일 생성 및 업데이트 |
| `patch-automation` | 4 | 크로스 플랫폼 복구를 위한 Ansible 플레이북 생성 및 검증 |
| `pentest-report` | 5 | 결과물 및 PATCH_LOG.md를 기반으로 전체 인게이지먼트 리포트 조합 |

## 에이전트

co-security 템플릿에는 6개의 전문화된 에이전트가 포함됩니다. 각 에이전트의 역할 정의는 `agents/<name>.md`에 있습니다.

| 에이전트 | 역할 |
|----------|------|
| **PM (보안 PM)** | 인게이지먼트 단계 오케스트레이션, 권한 부여 게이트 집행, 작업 순서 관리 및 이해관계자 보고 |
| **레드 팀 리드** | 공격 전략 정의, Phase 3 익스플로잇 감독, 교전 규칙 준수 보장 |
| **펜테스터** | 직접 익스플로잇, PoC 개발, 원시 결과물 문서화 |
| **위협 모델러** | STRIDE/PASTA 분석, 공격 트리 구성, 위험 우선순위 지정 |
| **패치 엔지니어** | 복구 스크립트 개발, Ansible 플레이북 작성, SSH를 통한 크로스 플랫폼 패치 배포 (Windows/macOS/Linux) |
| **리포트 작성자** | 기술 및 경영진 보고서 작성, CVSS 점수 산정, 복구 단계 문서화 |

에이전트는 네이티브 `Agent` 도구를 통해 디스패치합니다. 파일 경로를 참조하는 대신 관련 `agents/<name>.md` 내용을 프롬프트에 직접 삽입하세요 — 서브 에이전트는 부모 세션과 파일 시스템 컨텍스트를 공유하지 않습니다.

## 설정

### CLAUDE.md

Claude Code 동작을 제어합니다: 자동화 훅, 슬래시 커맨드 정의, 플랜 모드 규칙, 작업 추적 관례, Git/PR 표준. co-security `CLAUDE.md`는 세션 시작 시 및 공격 스킬 실행 전 권한 부여 게이트 검사를 추가로 집행합니다.

### docs/authorization.md

인게이지먼트를 위한 서명된 권한 부여 문서를 저장합니다. 이 파일은 `verify-authorization` 스킬에 의해 검사됩니다. 필수 필드: 스코프 해시, 권한 부여자(이름 및 역할), 인게이지먼트 기간(시작/만료일), 스코프 내 대상 목록. Phase 1 이후 활동 전에 이 파일이 존재하고 유효해야 합니다.

### docs/scope.md

대상 시스템, IP 범위, 애플리케이션 엔드포인트, 교전 규칙, 명시적 스코프 외 경계를 정의합니다. `engagement-scoping` 스킬을 사용하여 Phase 0에서 작성합니다. 이 파일은 모든 에이전트에 대한 권위 있는 참조 문서입니다 — 경계 밖의 모든 활동은 금지됩니다.

### PATCH_LOG.md

Phase 4 동안 적용된 모든 패치의 시간순 로그입니다. 각 항목에는 결과물 ID, 영향받는 시스템, 패치 방법(수동/Ansible 플레이북), 타임스탬프, 검증 상태가 기록됩니다. `pentest-report` 스킬은 최종 리포트를 조합할 때 이 파일을 읽습니다.

### ansible/

크로스 플랫폼 패치 배포에 사용되는 모든 Ansible 인벤토리 파일, 플레이북, 역할이 포함됩니다. 플레이북은 결과물 ID별로 이름이 지정됩니다(예: `playbooks/CVE-2025-1234.yml`). 프로덕션 대상에 적용하기 전에 항상 스테이징 인벤토리에서 플레이북을 테스트하세요.
