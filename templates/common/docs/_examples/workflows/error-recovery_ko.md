# 에러 복구 프로토콜

이 문서는 멀티 에이전트 워크플로우를 위한 에러 복구 프로토콜을 정의합니다.

## 개요

에이전트가 작업 실행 중 에러를 encountered하면, 사용자에게 에스컬레이션하기 전에 구조화된 복구 프로세스를 따릅니다.

## 에러 분류

에러는 4가지 범주 중 하나로 분류됩니다:

| 유형 | 설명 | 예시 |
|------|------|------|
| **도구 에러** | MCP 도구 또는 명령 실행 실패 | 파일을 찾을 수 없음, 권한 거부됨 |
| **컨텍스트 에러** | 정보가 부족하거나 모호함 | 요구사항 누락, 명세 불명확 |
| **논리 에러** | 에이전트 추론 또는 구현 버그 | 잘못된 패턴 선택, 잘못된 분석 |
| **외부 에러** | 타사 서비스 또는 종속성 실패 | API 타임아웃, 종속성 충돌 |

## 복구 프로세스

### 1단계: 에러 분석

에러가 발생하면 에이전트는 다음을 수행해야 합니다:

1. **에러 기록**: 에러 유형, 메시지, 컨텍스트 기록
2. **에러 분류**: 어떤 범주에 속하는지 결정
3. **근본 원인 파악**: 에러가 발생한 이유 이해
4. **복구 가능성 확인**: 자동으로 수정 가능한지 결정

### 2단계: 자동 복구

복구 가능한 에러의 경우 최대 **3회의 자동 재시도**가 가능합니다:

```typescript
interface RetryConfig {
  maxRetries: 3;
  initialDelay: 1000;      // 1초
  backoffMultiplier: 2;    // 지수 백오프
  maxDelay: 10000;         // 10초 상한
}
```

#### 에러 유형별 재시도 전략

| 에러 유형 | 재시도 전략 | 최대 재시도 |
|------------|---------------|-------------|
| 도구 에러 | 동일한 매개변수로 재시도 | 3 |
| 컨텍스트 에러 | 명확한 질문 후 재시도 | 2 |
| 논리 에러 | 접근 방식 수정 후 재시도 | 1 |
| 외부 에러 | 대기(지수 백오프) 후 재시도 | 3 |

### 3단계: 에스컬레이션

최대 재시도 후 자동 복구가 실패하면 사용자에게 에스컬레이션:

```json
{
  "escalation": {
    "task_id": "TASK-2025-001",
    "agent": "agent-name",
    "error": {
      "type": "tool | context | logic | external",
      "message": "에러 설명",
      "attempts": 3
    },
    "context": {
      "phase": "current-phase",
      "recent_actions": ["action1", "action2"]
    },
    "suggestions": [
      "제안 1",
      "제안 2"
    ],
    "requires_user_input": true
  }
}
```

## 에러 복구 템플릿

### 템플릿 1: 도구 에러 복구

```
## 도구 에러 감지됨

**도구**: tool-name
**에러**: error-message
**시도**: 1/3

### 복구 작업
- 동일한 매개변수로 재시도 중...
- 재시도 전 2초 대기...

[재시도 실패 시]
- 2/3 시도 실패
- PM에 안내 요청을 위해 에스컬레이션
```

### 템플릿 2: 컨텍스트 에러 복구

```
## 컨텍스트 에러 감지됨

**문제**: 진행에 필요한 정보가 부족함
**누락**: requirement-specification

### 명확화 필요
- [특정-경우]에 대한 예상 동작은 무엇입니까?
- 이 시나리오에서 시스템이 X를 처리해야 합니까 아니면 Y입니까?

[재시도 전 사용자 응답 대기 중]
```

### 템플릿 3: 논리 에러 복구

```
## 논리 에러 감지됨

**문제**: 분석 결과 모순되는 결론
**근본 원인**: [시나리오]에 대해 잘못된 패턴 적용

### 수정
- [올바른-패턴]을 사용하여 재평가 중
- 새로운 이해를 바탕으로 접근 방식 조정

[수정된 논리로 진행]
```

### 템플릿 4: 외부 에러 복구

```
## 외부 에러 감지됨

**서비스**: service-name
**에러**: connection-timeout
**시도**: 1/3

### 복구 작업
- 2초 대기(지수 백오프)...
- 서비스 연결 재시도 중...

[모든 재시도 실패 시]
- 3회 시도 후 외부 서비스를 사용할 수 없음
- 사용자에게 서비스 상태 확인 제안
- 수개입을 위해 에스컬레이션
```

## 스크립트와의 통합

`scripts/retry-handler.ts`는 자동화된 재시도 논리를 제공합니다:

```typescript
import { withRetry, escalateToHuman } from "./retry-handler";

const result = await withRetry(
  () => dispatchSubagent(task),
  {
    maxRetries: 3,
    initialDelay: 1000,
    backoffMultiplier: 2,
    maxDelay: 10000
  },
  "Task Description"
);

if (!result.success) {
  escalateToHuman(
    "Task Description",
    result.lastError!,
    result.attempts
  );
  process.exit(1);
}
```

## 모범 사례

1. **빠른 실패**: 명백히 복구 불가능한 에러의 경우 즉시 에스컬레이션
2. **컨텍스트 보존**: 재시도 간 전체 컨텍스트 유지
3. **점진적 개선**: 각 재시도는 이전보다 더 많은 정보를 가져야 함
4. **명확한 커뮤니케이션**: 사용자에게 실패 내용과 이유를 항상 설명
5. **학습**: 향후 참조를 위해 `memory/`에 에러 패턴 문서화

## 에러 예방

에러를 최소화하기 위해:

- **사전 확인**: 시작 전 모든 전제 조건 확인
- **점진적 검증**: 진행 전 각 단계 확인
- **명확한 계약**: 에이전트 통신을 위해 handoff-spec.md 사용
- **방어적 코딩**: 엣지 케이스를 명시적으로 처리
