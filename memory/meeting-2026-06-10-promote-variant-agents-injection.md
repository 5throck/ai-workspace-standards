## Transcript

### Round 1

**Architect**: (Round 1)

The core issue is that `generate-variant.ts` has no `generateAgentsMd()` function at all. While `generateClaudeMd()` and `generateGeminiMd()` exist, AGENTS.md is either copied as-is from the L2 source via `manifest.keepInVariant` fallthrough or not generated at all. With the new 6 `VARIANT-*` placeholders added to L1 AGENTS.md, the pipeline must fill them with variant-specific content at generation time.

My proposal is a structured, data-driven approach: read L1 AGENTS.md as base, then call `injectVariantPlaceholders(content, metadata)` to replace each `<!-- VARIANT-*-START/END -->` block with content generated from `metadata.agentRoster`. This maintains the single source of truth principle since `variant.json`'s `agents[]` array already holds the agent data.

I also note that `AgentDefinition` currently only has `name`, `tier`, `model`, `description` — it lacks `dispatchTriggers`, `phasePrimary`, `writeScope`, `parallelizable` fields needed to fully populate the Phase Gate, Role Boundary, and Dispatch Trigger sections.

**Automation-Engineer**: (Round 1)

I agree with Architect's structural approach over regex substitution. The 6 placeholders have different structural requirements — table rows, dispatch trigger lists, role boundary rules — so a single regex handler would be fragile and hard to maintain.

However, I want to flag a practical concern about the AgentDefinition extension. If we add `dispatchTriggers`, `phasePrimary`, etc. as required fields, existing usages of `AgentDefinition` across scan-l2-project.ts and reconcile-with-l0-l1.ts will need updates. I recommend making them all optional (`?`) and having `scan-l2-project.ts` extract them from agent frontmatter where available.

For the implementation structure:
```
generateAgentsMd(variantPath, metadata, manifest)
  ├── read L1 templates/common/AGENTS.md as base
  ├── injectVariantPlaceholders(content, metadata)
  │    └── replace each VARIANT-*-START/END block
  └── write to variantPath/AGENTS.md
```

The `injectVariantPlaceholders` function should be a standalone utility, not embedded in `generateAgentsMd`, since it may be reused later for other placeholder injection scenarios.

**Docs-Writer**: (Round 1)

From a documentation perspective, I need to flag that `l2-to-variant-conversion-pipeline.md` §5.5.2 (AGENTS.md Registration Validation) is written against the L0 workspace AGENTS.md — it checks whether L2 agents are registered in the workspace-root AGENTS.md, which is no longer the right model. The new model is: L1 AGENTS.md provides the template with placeholders, and L2 AGENTS.md is generated by filling those placeholders.

This design doc needs two changes:
1. §5.5.2 rewritten to reflect the L1 placeholder-based check (verify placeholders are filled, not L0 registration)
2. A new §5.5.4 section documenting the AGENTS.md injection design: the 6 placeholder names, `injectVariantPlaceholders` function contract, and `AgentDefinition` field extensions

I also want to note that `pm-md-variant-specific-content-injection-design.md` currently only covers pm.md injection. Since AGENTS.md now follows a similar placeholder injection pattern, a parallel section in that doc makes sense for consistency.

### Round 2

**Architect**: (Round 2)

Responding to automation-engineer's point about making fields optional — fully agreed. The interface change should be backward-compatible. For `scan-l2-project.ts`, the extraction logic should read standard frontmatter fields from agent `.md` files:
- `dispatch_triggers:` → `dispatchTriggers`
- `phase:` → `phasePrimary`
- `write_scope:` → `writeScope`
- `parallelizable:` → `parallelizable`

Where these fields are absent, generate reasonable defaults with a `<!-- TODO: customize for [agent-name] -->` marker inside the placeholder block so the generated AGENTS.md is immediately usable but clearly signals what needs manual refinement.

The key design decision for `injectVariantPlaceholders`: it should handle the case where `metadata.agentRoster` is empty gracefully — leave placeholder comments intact (no-op) rather than generating empty table sections, which would fail downstream validation.

**Automation-Engineer**: (Round 2)

One implementation detail worth resolving: the 6 placeholder blocks have different output structures:

| Placeholder | Generated content shape |
|-------------|------------------------|
| VARIANT-AGENTS | Table rows (one per agent) |
| VARIANT-AGENT-DETAILS | Subsections (one H3 per agent) |
| VARIANT-DISPATCH-TRIGGERS | Table rows with trigger keywords |
| VARIANT-PHASE-GATE | Table rows mapping deliverable → agent |
| VARIANT-SUBAGENT-ROSTER | Table rows with parallelizable/write scope |
| VARIANT-ROLE-BOUNDARY | Table rows mapping scenario → use/do-not-use |

Each needs its own generator sub-function. I propose a `PlaceholderGenerator` map pattern:
```typescript
const generators: Record<string, (agents: AgentDefinition[]) => string> = {
  'VARIANT-AGENTS': generateAgentRosterRows,
  'VARIANT-AGENT-DETAILS': generateAgentDetailSections,
  ...
};
```

This makes it easy to add new placeholder types in the future without modifying `injectVariantPlaceholders` itself.

**Docs-Writer**: (Round 2)

I support automation-engineer's `PlaceholderGenerator` map approach — it also makes the design document cleaner to specify, since each generator function can be documented independently.

For the design doc updates, I'll structure §5.5.4 around: (1) overview of 6 placeholders, (2) `injectVariantPlaceholders` function contract, (3) `PlaceholderGenerator` map pattern, (4) `AgentDefinition` extension fields, (5) empty roster handling policy. This gives automation-engineer a clear spec to implement from.

One final point: the docs should clarify that AGENTS.md injection happens at **variant generation time** (Wave 3 / generate-variant.ts), not at promote time (the promote step promotes the already-generated variant). This distinction avoids confusion with pm.md which uses a different mechanism (extends: YAML chain resolved by merge-frontmatter.ts at project scaffold time).
