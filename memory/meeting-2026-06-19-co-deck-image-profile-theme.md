**[research]** (Round 1)

**[html-build]** (Round 1)

storyline identified the core issue precisely. The single HTML format was a stopgap from the start and needs to be extended for different lecture styles.

Proposed **HTML format registry** structure:

```
templates/co-deck/html-themes/
├── classic/          # current format (left: text, right: image)
│   ├── theme.json
│   └── slide-renderer.js
├── minimal/          # text-focused, clean layout
├── visual-heavy/     # full-screen image + overlay text
└── academic/         # 2-column academic style
```

Each theme declares rendering rules per slide type via `theme.json`. The `html-build` agent reads the `theme` field from `lecture-profile.md` to select the theme.

This structure is directly linked to **lecture profile integration** — including theme settings in the profile allows "this lecture always uses minimal theme" type of configuration.

**Image integration**: images downloaded by `image-curator` are stored in `presentations/<project>/assets/images/`, and HTML generation binds per-slide image paths.

Question to design: image aspect ratios and placement rules differ by theme — how should this be integrated with `layout_spec.json`?

---

**[design]** (Round 1)

html-build's theme registry idea is the right direction, but one important design principle must be established first: **themes should be a set of layout rules, not completely independent codebases.**

`layout_spec.json` is measured data generated by `measure-layout.ts`. If different HTML is rendered per theme, measured values differ per theme and PDF coordinate mapping in `gen-slides-pdf.ts` breaks.

Proposed approach:

```
html-themes/
├── base/              # shared CSS variables + slide structure
└── overrides/
    ├── classic.css    # current style
    ├── minimal.css    # variable overrides only
    └── visual-heavy.css
```

CSS variable override approach preserves:
- Slide DOM structure → no need to re-run `measure-layout.ts`
- Each theme only changes color/font/margin variables
- `layout_spec.json` coordinate values remain valid across themes

Image placement controlled by CSS Grid / Flex ratios. `visual-heavy` theme: `--image-panel-width: 100%`, `classic`: `--image-panel-width: 45%`.

Question to architect: from an L2 layer design perspective, what criteria determine what should be declared in `variant.json`'s `agent_manifest` or `script_manifest`?

**Classification of new additions:**

**`image-curator` agent design principles:**
- Tools: Web Search MCP (`mcp__web-search-prime`) + Bash(curl/wget)
- Image source priority: Unsplash API → Pexels API → Google Images (license filter)
- Output: `presentations/<project>/assets/images/<slide-id>.<ext>` + `image-manifest.json`
- **Critical constraint**: only copyright-clear sources, commercial license required

**`lecture-profile.md` required fields:**
```yaml
---
title: Lecture Title
audience: graduate | undergraduate | practitioner
level: intro | intermediate | advanced
theme: classic | minimal | visual-heavy | academic
language: ko | en
keywords: [keyword1, keyword2]
---
```

**Image acquisition timing**: calling `image-curator` after research → storyline completion is correct. `image_query` per slide can only be finalized after the storyline is confirmed.

**[html-build]** (Round 2)

Adopting design's CSS variable override approach. The immutable DOM structure principle is essential for PDF coordinate consistency.

**Theme switching implementation:**

```html
<!-- Theme links injected in HTML header -->
<link rel="stylesheet" href="../../html-themes/base/base.css">
<link rel="stylesheet" href="../../html-themes/overrides/{{ theme }}.css">
```

`html-build` agent reads `theme` value from `lecture-profile.md` and dynamically sets the second link. Default `classic` if no theme specified.

**`theme_manifest` to add to `variant.json`:**
```json
"theme_manifest": {
  "themes_dir": "html-themes",
  "available": ["classic", "minimal", "visual-heavy", "academic"],
  "default": "classic"
}
```

`validate-templates.ts` validates `themes_dir` existence as a B-04 check.

---

**[design]** (Round 2)

Confirming adoption of CSS variable override approach. Clarifying visual differentiation points per theme:

| Theme | Primary Use | Image Ratio | Font Characteristics |
|-------|------------|-------------|---------------------|
| classic | current default, general purpose | 45% right panel | medium |
| minimal | text-heavy lectures | none or 20% | large, readability-first |
| visual-heavy | visual storytelling | 70-100% background | overlay white |
| academic | thesis / academic presentations | 30% illustration | small, high density |

**Integration with image-curator**: `visual-heavy` theme requires full-background images, so it only applies to slides with `image_role: background`. `minimal` theme must work without images.
