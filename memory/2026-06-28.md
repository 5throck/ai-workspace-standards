## Decisions

| Item | Decision |
|------|----------|
| Roadmap | Full 3-tier roadmap, incremental implementation |
| Tier 1 (immediate) | G1 (Fullscreen API) + G2 (@media print) |
| Tier 2 approach | B: Renderer Adapter pattern |
| Tier 3 order | E8 (CSS tuner) → E6 (Dual-Screen) → E1 (HMR) |
| G3 (Grid Overview) | Deferred to post-Tier 2 |

### Tier 1: Foundation (immediate scope)

| # | Feature | Files | Complexity |
|---|---------|-------|------------|
| G1 | Fullscreen API — F key + footer toggle button | ppt-engine.js, ppt-engine.css | 🟢 Low |
| G2 | @media print — Ctrl+P renders one slide per page | ppt-engine.css | 🟢 Low |

### Tier 2: Fidelity (next)

| # | Feature | Description |
|---|---------|-------------|
| 2a | ppt-engine.js/css load + init() in preview.html |
| 2b | Dummy slideData → ppt-engine format conversion |
| 2c | Per-theme preview-renderer.js (Renderer Adapter pattern) |
| 2d | Build pipeline integration (optional) |

### Tier 3: Advanced UX (future)

| # | Feature | Description |
|---|---------|-------------|
| E8 | CSS Variable Tuner — real-time accent/color/font customization |
| E6 | Presenter Dual-Screen — BroadcastChannel sync |
| E1 | HMR — WebSocket + fs.watch for live reload |

### Key Architecture Decision: Renderer Adapter (Tier 2)

- Each theme gets a `preview-renderer.js` with common interface: `renderSlide(data, container)`
- preview.html dynamically loads the selected theme's adapter
- Balances fidelity with maintainability
- 5 themes × 1 adapter each = manageable maintenance burden
