# Decision Record: Fix Variant Scaffolding Validation Failure

- **Date**: 2026-06-08
- **Topic**: Fix variant scaffolding validation failure
- **Status**: Approved

## Context
Creating new projects with `new-project.sh` failed in the "Validating template integrity..." phase due to missing `CLAUDE.md` and `GEMINI.md` in variant template directories (e.g., templates/co-develop, templates/co-design).

## Analysis
Individual variant templates do not contain `CLAUDE.md` or `GEMINI.md` at their root by design. Instead, they inherit them from `templates/common` at project creation time, and only override them if required. `template-validation.ts` mistakenly listed them as mandatory for all variants.

## Decision
Removed `CLAUDE.md` and `GEMINI.md` from `variantRequired` in `scripts/helpers/template-validation.ts`, and bumped its version from `1.0.0` to `1.0.1`. Update version registry accordingly.

# Decision Record: Fix extends path security validation and documentation false positives during new project scaffolding

- **Date**: 2026-06-08
- **Topic**: Fix extends path security validation and documentation false positives during new project scaffolding
- **Status**: Approved

## Context
Creating new projects with `new-project.sh` failed in the post-scaffold audit phase. The extends validation failed for `co-develop/agents/pm.md` because `extends: ../../common/agents/pm.md` was resolved relative to the copied project directory rather than the template source directory. Additionally, the extends verification flagged an unresolved extends field in `docs/variant-pm-spec.md` due to an unindented example code block.

## Decision
1. Updated `merge-frontmatter.ts` (v1.1.1) to support a 3rd CLI argument `original-context-path`. When provided, the validator validates this path instead of the copied target path.
2. Updated `new-project.sh` (v1.4.7) to pass the template source path (`$src_file`) as the 3rd argument to `merge-frontmatter.ts`.
3. Indented the code block in `templates/common/docs/variant-pm-spec.md` to avoid matching `^extends:`.

# Decision Record: Implement remove_sections in merge-frontmatter.ts

- **Date**: 2026-06-08
- **Topic**: Fix variant pm.md content — `remove_sections` not applied during scaffolding
- **Status**: Approved

## Context
When scaffolding new projects with `new-project.sh`, the generated `agents/pm.md` in variant projects retained all L0 workspace-specific sections (e.g., `## Governance Workflow`, `## Updated Role`, `## Agent Roster`, `## Dispatch Protocol`, `### Phase Determination`) even though the L1 template's frontmatter specified `remove_sections` to strip them. This caused variant projects to carry governance instructions intended only for the L0 workspace root.

## Root Cause
`merge-frontmatter.ts` correctly merged frontmatter YAML via the `extends` chain but never applied the `remove_sections` list to the markdown body content. The `remove_sections` key existed in the merged frontmatter but had no effect on the output body.

## Decision
1. Added `removeSections()` function to `scripts/helpers/merge-frontmatter.ts` (v1.2.0).
2. After resolving the extends chain and building the final body content, the function now strips all sections listed in `remove_sections` from the markdown body.
3. Uses **prefix-based heading matching** (case-insensitive) to handle headings with suffixes (e.g., `## Updated Role` matches `## Updated Role (Phase 0/1-2/5/6 Only)`).
4. The `remove_sections` key is deleted from the emitted frontmatter so it does not appear in generated files.
5. Synced updated `merge-frontmatter.ts` to `templates/common/scripts/helpers/merge-frontmatter.ts`.
6. Updated version registries in `scripts/SCRIPTS.md` and `templates/common/scripts/SCRIPTS.md`.

# Decision Record: resolve validate-templates.ts CLAUDE.md/GEMINI.md common blocklist conflict & fix pm-md-parser.ts variant_overrides type validation

- **Date**: 2026-06-08
- **Topic**: resolve validate-templates.ts CLAUDE.md/GEMINI.md common blocklist conflict & fix pm-md-parser.ts variant_overrides type validation
- **Status**: Approved

## Context
`validate-templates.ts` failed due to forbidden files (`CLAUDE.md`/`GEMINI.md`) in L1 `templates/common/`, which were consolidated in L1 per ADR-0033. At the same time, `audit.ts` flagged L2 variants as missing `CLAUDE.md` and `GEMINI.md`. In addition, `audit.ts` warned that `variant_overrides` parsing failed for L2 `pm.md` files because `pm-md-parser.ts` only validated string types for override keys, whereas the new v1.5.0 layout architecture uses structured objects/arrays.

## Decision
1. Removed `CLAUDE.md` and `GEMINI.md` from `forbiddenFiles` in `validate-templates.ts` (Check 0).
2. Removed `CLAUDE.md` and `GEMINI.md` from `requiredFiles` for L2 variants in `audit.ts`.
3. Redefined types and validated properties in `helpers/pm-md-parser.ts` to support both string and object/array types for `role`, `updated_role`, `governance_workflow`, `agent_roster`, and `dispatch_protocol`.
4. Bumped versions of modified scripts and registries: `validate-templates.ts` (v1.5.4), `audit.ts` (v2.6.2), `helpers/pm-md-parser.ts` (v1.0.2).
5. Ran `publish-to-template.ts` to propagate changes to common templates.
